# 시간복잡도 계산 in DFS, List(동적할당)

## 배운 것

### DFS에서의 시간복잡도 계산

DFS 탐색의 시간복잡도 = $O(V + E)$

**기본 원리:**
- 양방향 그래프, 인접 리스트에서의 경우 각 정점을 탐색하는 경우
  - 각 정점을 탐색하는 횟수의 합 = $V$
  - 각 정점에서 뻗어나가는 노드를 탐색하는 경우의 합 = $2E$
  - Big-O 함수에서는 상수를 무시하기 때문에 결과적으로 $O(V + E)$

**주의: 최악의 경우**
- 완전 그래프(complete graph)에서는 시간복잡도가 $O(V^2)$가 된다
- 완전 그래프에서는 간선(edge)의 개수가 $\frac{V(V-1)}{2} \approx V^2$
- 따라서 $O(V + E) = O(V + V^2) = O(V^2)$

> **핵심**: 그래프의 구조에 따라 같은 $O(V+E)$ 형태도 최악의 경우 $O(V^2)$까지 악화될 수 있음

### 리스트 동적할당에서 값 추가 시 시간복잡도 계산

**list.append()의 실제 시간복잡도:**

대부분의 경우: $O(1)$  
하지만 미리 할당해 둔 배열의 크기를 초과하면:
- 새로운 배열(기존 크기 × 2)을 재할당
- 기존 값들을 모두 복사
- 이 연산은 $O(N)$이 됨

**예시: 빈 리스트에 81번 append**

```
초기 할당: 5칸
메모리 확장 과정:
  5칸 → 10칸 (5개 복사)
  10칸 → 20칸 (10개 복사)
  20칸 → 40칸 (20개 복사)
  40칸 → 80칸 (40개 복사)
  80칸 → 160칸 (80개 복사)

총 연산수 = 81(순수 append) + 5+10+20+40+80(복사)
         = 81 + 75 + 80
         ≈ O(3N) 미만
```

**결론:**
- append()의 연산횟수 = **Amortized(분할상환) $O(1)$**
- 여러 번의 append를 평균내면 $O(1)$에 가까움

> **핵심**: 개별 연산은 $O(N)$일 수 있지만, 충분히 많은 연산을 평균내면 $O(1)$이 됨

### 연결리스트(Linked List)

**특징:**
| 연산 | 시간복잡도 | 이유 |
|------|----------|------|
| append (맨끝에 추가) | $O(1)$ | 마지막 노드 포인터만 변경 |
| search (특정 값 찾기) | $O(N)$ | 처음부터 순회해야 함 |
| delete (특정 노드 삭제) | $O(N)$ | 노드를 찾은 후 삭제 |
| **index access (인덱싱)** | ❌ **불가능** | 메모리에 연속 할당 안 됨 |

**연결리스트가 유리한 경우:**
-  조회(search)가 적을 때
-  전체를 다 순회해야 할 때
-  빈번한 추가/삭제가 있을 때 (배열과 달리 O(1))

>  **핵심**: 인덱싱이 필요 없으면 연결리스트, 빠른 접근이 필요하면 배열

## 막혔던 것 / 해결

### 1. 파이썬 list의 메모리 할당 원리

**막혔던 점:**
- 동적할당이 어떻게 이루어지는지 추상적이었음
- "메모리를 예시로 설명"해도 직관적 이해 어려움

**해결:**
- 강사님이 **힙(heap) 구조**로 메모리 안에서 변수 할당이 이루어지는 과정을 설명
- 구체적인 크기 변화(5→10→20→40...)를 보며 이해

---

### 2. DFS 시간복잡도의 "1차 vs 2차" 혼동

**막혔던 점:**
- $O(V + E)$는 결국 1차 아닌가?
- 왜 간선의 수에 따라 2차가 될 수 있는가?

**해결:**
- **핵심 깨달음 1**: 양방향 그래프에서 간선 탐색은 **2E번** 진행
- **핵심 깨달음 2**: 완전그래프에서 $E = \frac{V(V-1)}{2} \approx V^2$
  - 따라서 $O(V + V^2) = O(V^2)$
- **핵심 깨달음 3**: $O(V + E)$는 "1차"가 아니라 **"간선과 정점의 합"**
  - 같은 형태여도 그래프 구조에 따라 실제 복잡도는 달라짐

> **새로운 관점**: 빅오 표기법 안에서도 세세한 시간복잡도 계산이 중요함
